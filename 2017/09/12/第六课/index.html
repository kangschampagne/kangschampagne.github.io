1.需要处理一些登入态续登的问题

为什么一定要做数据缓存

server render 页面加载速度最快
加载顺序
HTML download -> CSS download -> JS download -> parase js -> run js -> ajax => render
做法1. 省掉 ajax请求使用 localstorage 等
100 request -> DB -> 1 request backend
做法2. HTML -> JSONP -> CSS -> JS -> JS parse -> JS run -> render
ajax -> 100 ~ 200ms 通过 DB 使用缓存

合并请求 -> graphQL http://graphql.org/learn/

homework 3

// 0. 默认，情况下，如果没缓存，成功请求回调一次，如果有请求成功缓存应当回调两次，这样可以保证页面快速渲染，而不需要等数据返回
场景：先读缓存 渲染再 请求新数据

// 1. lazy更新，有缓存的情况下直接读取缓存，只有一次回调，但会发出请求，请求返回后只更新缓存里的数据为最新的，这样，下次再调用就能拿到这次更新的数据了
options.lazy = true
场景：一般加 maxAge ，过去时间点的 订单或者 过去时间点 的 销量数据 定时任务

// 2. 在maxAge内使用缓存，则当发现缓存里的数据小于maxAge，则直接读取缓存，不发出任何请求，否则发出请求读取请求的数据
options.maxAge = 1000 // 即缓存是1s，也就是说这个单位是ms
场景： 高频率的请求 F5 service render memcache redis

数据的存储一般是带有版本号信息。
例如

Key => refly1✂\$stranger\$✂ppp://123?✂1504940051878
Value => { “r”: { “ret”: 123 }, “v”: 1 }

版本号 v 为 数据结构的 版本号，即数据结构不同，避免因为数据结构更改导致抛错。
数据等级 refly 1 => 等级 1

问： 大家觉得localStorage是纯单key好还是存多key好？
多 key 比较好， 因为单个 key 的读取速度较慢。
性能测试点。

在localStorage 操作要比在 内存上操作要慢很多。

不能用 cookie 做缓存， 每次都会有请求。
